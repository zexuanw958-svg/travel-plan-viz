# 联网调研指南

生成页面前，用你所在 Agent 的**联网搜索 / 网页抓取工具**补全数据（如 Claude Code 的 WebSearch/WebFetch、Codex 的联网能力等）。**不查实时票价/机票价格**（易过期出错）。

## 第三方 skill 适配（可选增强，有则用、无则降级）

调研开始前，先判断**当前 Agent 是否可调用相关官方旅行类 skill / MCP**（用户自行安装的，如飞猪、高德、腾讯地图、滴滴等）。**有就优先用它们拿对应数据，没有就走本指南后面的静态调研**——两条路产出同一套 `trip` 数据结构，成品不因缺这些 skill 而残缺。

下表为各家公开能力盘点（2026-06 调研一手开放平台文档，**以各家官方文档实时为准**，能力会变）：

| 用户若装了 | 可优先用它拿（对应咱们哪个功能） | 备注 |
|------|------|------|
| **高德地图** MCP/skill（增强面最大，多数工具开箱即用） | **地理编码 → 精确坐标**（替代"搜经纬度"猜）；**公交/驾车/步行/骑行路径规划 → 点到点交通**（实时路况+多方案）；**天气查询 → 行前须知天气**；**POI 关键词/周边/详情搜索 → 景点&酒店信息**；**生成专属地图 → 整条行程唤端地图链接**；**导航/打车 → actionLink** | 调研到 15 个工具，是命中咱们薄弱点最多的一家 |
| **飞猪 flyai** skill/MCP（实时直连官方库，结果可真实预订） | **机票搜索 → 航班候选**；**酒店 → 酒店片区**；**景区门票/度假 → 门票**；均返回**真实预订链接**作 actionLink | **需 API Key**；体验版数据有缺 |
| **腾讯位置服务** skill/MCP（能力与高德重叠，开发向更重） | 路线规划、POI 搜索（8000 万+）、地址服务、实时路况导航；另有 3D/室内地图（一般用不上） | 多与高德二选一即可 |
| **滴滴** MCP/skill | **打车全流程**（叫车/预估/跟踪/取消）→ 交通的用车方案与**叫车唤端链接**；也有驾车/公交/步行/骑行路线规划 | Beta 版返回跳转链接 |

没装任何一家时，全部走本指南后面的静态调研，成品不缺区块。

**坐标系陷阱（用高德/腾讯必读）**：高德与腾讯地图返回的坐标是 **GCJ-02**（国测局加密坐标），而本 skill 的地图用 OSM 瓦片（**WGS-84**）。直接把 GCJ-02 坐标写入 `trip`，中国境内点位会系统性偏移一百到几百米——恰好制造你想避免的"坐标错位翻车"。**凡来自高德/腾讯的坐标，写入前必须用 `assets/map.js` 的 `gcj02ToWgs84(lat, lng)` 转换**（境外坐标该函数会原样返回，多转不会错）。静态调研（维基/搜索）拿到的坐标一般已是 WGS-84，不用转。

**具体示例（点到点交通）**：本指南的「点到点交通」默认是静态调研出一个方式+票价+耗时。若用户装了高德/腾讯地图 skill，改用它的**路线规划**（高德有公交/驾车/步行/骑行四种，公交含火车/地铁/公交综合方案），拿到结合**实时路况**的多方案（更科学合理），并把官方返回的导航链接作为该节点的 `actionLink`。另一个高价值点：用高德**地理编码**拿精确坐标，比"搜名称+经纬度"可靠得多（坐标错位是地图最常见的翻车点）。

**三条铁律（破了就违背架构红线）**：
1. **责任边界**：用官方 skill 拿来的数据，其**实时性与真实性由该官方 skill 负责**，我们只做适配编排，不自行复核也不背书。
2. **标注来源 + 中性措辞**：凡用官方 skill 拿的数据，在页面对应区块注明数据来源（如「实时数据来源：高德地图 skill，时效性由其保证」），与 AI 静态整理的内容（标"参考·可能过时"）区分开；**用中性工程措辞，不夸、不推荐、不替任何一家打广告**。
3. **行动链接只用官方返回的**：`actionLink` 的 URL **优先用官方 skill 返回的链接**（带正确参数最稳）；**不要手拼 deeplink / URL scheme**（易失效）。官方 skill 不在场时，宁可不加行动链接，也别放可能打不开的手拼链接。

> 本 skill 自身永远不自行抓实时票价——实时数据只可能来自用户已装的官方 skill。详见 page-contract 的 `dataSources` 与节点级 `actionLink` 字段。

## 每个景点/酒店需采集

- 坐标（lat/lng）：搜索"<名称> 经纬度 / coordinates"，或从地图类结果取。
- 真实图片 URL（**务必能加载**）：
  - **不要手拼** `upload.wikimedia.org/.../thumb/X/XX/...` 这种带哈希前缀的直链——哈希猜错就 400/404。
  - 用稳健形式：`https://commons.wikimedia.org/wiki/Special:FilePath/<URL编码的文件名>?width=600`，它会自动重定向到正确缩略图，无需哈希。文件名取自该图在 Commons 的真实文件名。
  - **每个图片 URL 都要 `curl -s -o /dev/null -w "%{http_code}" -L <url>` 验证返回 200 才用**；非 200（文件不存在等）就换一张真实存在的图，实在没有就留空走 onerror 降级，绝不放已知打不开的链接。
- 评分：来自点评类来源的概览分（注明是概览，非实时）。
- 一句话点评：综合调研后用一句中文概括亮点。
- 营业时间 + 休息日（如"12:30–22:30，周日休"）：搜索"<名称> 营业时间 opening hours"。
- 门票参考价（如有）：给区间或单价，注明为参考、非实时。

## 行前须知（按出发日期/季节定制）

- 天气：搜索"<城市> <出发月份> 天气 平均气温 降雨"，给气温区间、降雨规律（如午后雷暴）、台风/极端天气季节提醒。
- 穿搭：据天气给具体建议（含室内空调偏冷等本地常识）。
- 支付：当地主流支付方式（如香港八达通）、是否需备现金。
- 必备 App：交通、天气、热门景点排队等。
- 购票时机：热门景点建议提前几天网上购票。

## 点到点交通

- 为关键行程点给出"如何到达"：方式（地铁/巴士/缆车/轮渡/步行）、参考票价、耗时。搜索"<A> 到 <B> 怎么走 交通"。机场往返单独给（如机场快线 vs 巴士的价/时对比）。

## 时令限定活动

- 搜索"<城市> <出发月份> 活动 / 限定 / 展览"，标注起止日期；对应景点 slot 写入 `seasonal`。

## 天气驱动排程

- 若目的地该季节有明显午后高温/雷暴/严寒等特征：把户外项目排在凉爽时段（上午、傍晚），午后/极端时段安排室内（商场、博物馆、乐园），并在 `tips` 与当日 `tips` 中说明。

## 餐饮（每餐给必点菜 + 参考价）

- 每天给 1–2 餐具体推荐：餐厅名 + 营业时间/休息日 + 2–3 个必点菜及参考价。搜索"<城市> <区域> 必吃 推荐 招牌菜"。

## 每日当地美食

- 搜索"<城市> 必吃 / 当地特色美食"，每天给 1 条具体推荐。

## 航班（待选，不查实时价）

- 用户未预订时，**给 3-5 个待选班次**（不要只给一个，万一订不上有备选）。
- 搜索"<出发地>到<目的地> 航班 航空公司"，给真实存在的航司+航班号+大致时刻；`note` 写机型/是否直飞/经停/大致价位区间（区间即可，不查实时价）。
- 已预订的（用户告知）单独高亮，标"已预订"。

## 酒店（按片区 + 价位推荐，不指定单一酒店）

- 先看本次行程各景点的地理分布，**综合选 1-2 个建议住宿片区**（如香港多落在尖沙咀/中环/铜锣湾），写明推荐理由（离哪些点近、交通如何）。
- 每个片区按 `经济 / 中档 / 高端` 三档各给 1 个真实存在的代表酒店：名称 + 大致每晚价位区间 + 一句话说明（位置/特点）。价位为区间参考，非实时房价。

## 免责声明（必给，覆盖全部信息）

- **所有联网整理的信息**（天气、航班、酒店、餐厅、景点、门票、价格、营业时间、评分、活动等）都可能不准或已过时，不只是机票/酒店。页面必须给出覆盖全部信息的免责声明，提示用户到官方/订票订房/地图等 App 核实后再决定。尽量给真实可信信息，但不承诺与实时一致。
- **若部分数据来自用户已装的官方 skill**（见上节适配）：该部分实时性/真实性由对方官方 skill 负责，本页只做适配呈现、不另行背书；免责声明里照样提示以对应官方渠道为准即可（无需把责任揽到自己身上）。

## 提前预订项与提前天数（喂给 reminders）

按经验 + 联网确认，标记需提前订的项目并给 `leadDays`：

| 类型 | 建议提前天数 | 说明 |
|------|------------|------|
| 机票（国内） | 30 | 越早越便宜 |
| 机票（国际） | 45 | 含签证缓冲请另列 |
| 跨城高铁 | 15 | 放票后尽早 |
| 热门景点/限流（如长白山、故宫） | 7 | 官方放票窗口 |
| 网红餐厅/演出 | 3 | 需预约 |
| 酒店（旺季） | 30 | 旺季房紧 |

- 实际数值应结合目的地与季节联网核实后微调，并在对话里向用户说明依据。
