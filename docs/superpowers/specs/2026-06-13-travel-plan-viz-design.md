# 旅行计划可视化 Skill — 设计文档

- 日期：2026-06-13
- Skill 名称（暂定）：`travel-plan-viz`（旅行计划可视化）
- 状态：设计已确认，待写实现计划

## 一句话

把一趟旅行变成一个美观、可离线、手机优先的**单文件 HTML** 页面，内含交互地图、每日时间轴、出发前订票提醒。灵感来自社区 "vibe coding" 玩法，升级为正式可复用的 Claude Code skill。

## 两种入口

两种入口最终都汇聚到同一个"生成 HTML"步骤。

- **模式 A · 从零规划**：用户给目的地 + 天数（例 "香港 4天3晚"）→ skill 联网调研后产出详实的逐日计划（先在对话里以 markdown 呈现）→ 用户来回调整 → 用户确认后生成 HTML。
- **模式 B · 已有计划**：用户提供现成计划（纯文字，或一个旧的 HTML 文件）→ skill 解析 → 直接进入生成步骤（仍会联网补充图片 / 坐标 / 评分等缺失数据）。

## 架构：混合（固定引擎 + 灵活美学）

错误率高的"机械逻辑"做成固定可复用资产；视觉表现交给 frontend-design 每次重生。

### 固定复用的"引擎"（skill 自带，注入到产出 HTML 中）

- `map.js` — Leaflet + 免费 OSM 瓦片（无需 API key）；编号标记；按行程顺序连虚线路线；点击弹出名字 + 时间；每个点附"导航"深链（`geo:` / 高德 / Google Maps）。
- `reminders.js` — 输入出发日期 + `{事项, 提前天数}` 列表 → 计算每个截止日期 → 渲染两处：①页顶"出发前待办清单"（按截止日期排序）②每日时间轴上对应项目的 ⚠️ 徽标。两处由同一份数据生成，互相呼应。
- `page-contract.md` — 内容结构契约，向 frontend-design 说明每个区块需要哪些数据字段。

### 每次重生的（交给 frontend-design）

整体布局、配色（每趟行程不同以便区分）、字体、卡片与时间轴样式、整体精修。

## 数据来源（联网，票价不实时）

- 用 WebSearch / WebFetch 获取：景点/酒店的真实图片 URL、评分、一句话点评、坐标（经纬度）、每日当地美食推荐。
- 票价 / 机票：**不查实时价格**（避免过期 / 错误数据）。改为标记"通常需提前预订"的项目（如长白山门票、热门景点、跨城高铁 / 机票），赋予推荐提前天数 → 喂给 `reminders.js`。
- 图片使用在线缩略图 URL（联网时加载；离线可对整页截图存档，符合参考玩法）。除图片外保持单文件自包含。

## 产出页面结构（单 HTML 文件，手机优先）

1. 页顶：行程标题 + **出发前待办清单**（带截止日期，按 deadline 排序）
2. 航班 / 酒店高亮区（尤其是已预订的）
3. **交互地图**（编号点 + 虚线路线 + 导航链接）
4. **每日时间轴**（早 / 中 / 晚）；每个景点卡片含真实图片 + 评分 + 一句话点评；需提前预订的项挂 ⚠️ 徽标
5. 每日当地美食推荐

> HTML 本身即 context：用户之后可将文件丢回，说"第三天太赶，把 X 挪到第四天"，在原结构上修改。

## 文件结构

```
travel-plan-viz/
  SKILL.md              # 流程：判断模式 → (A) 规划 + 迭代 / (B) 解析 → 联网调研 → 生成
  assets/
    map.js              # Leaflet 引擎
    reminders.js        # 截止日期计算 + 渲染
    page-contract.md    # 给 frontend-design 的内容 / 数据契约
  references/
    research-guide.md   # 如何调研图片 / 坐标 / 评分 / 美食 + 提前预订启发式
```

## 工作流（SKILL.md 逻辑）

1. 判断模式（用户是否已有计划）。
2. 模式 A：联网调研目的地 → 在对话中提出详实逐日计划 → 迭代直到用户确认。
3. 联网采集 / 补全数据（坐标、图片、评分、美食、需提前预订项及提前天数）。
4. 调用 frontend-design 产出风格化 HTML，按 `page-contract.md` 接入 `map.js` 与 `reminders.js`。
5. 保存单个 HTML 文件；告知用户如何"把文件丢回来"进行后续修改。

## 测试

用样例行程（香港 4天3晚）跑一遍，浏览器打开验证：

- 地图渲染出编号点与虚线路线
- 每日时间轴内容正确
- 给定出发日期后提醒（待办清单 + 徽标）日期计算正确
- 导航深链可点击跳转
- 页面为单文件、可离线、手机端显示正常

## 不做（YAGNI）

- 实时票价 / 机票价格查询
- 代用户预订
- 多语言 UI
- 服务端 / 后端——全程静态单文件

## 实现后演进（截至 2026-06-14）

本设计完成后，经多轮使用反馈，skill 的内容契约在原始设计基础上扩展了以下能力（以 `travel-plan-viz/assets/page-contract.md` 为当前权威）：

- **航班改为待选多班次**：未预订时给 3–5 个真实候选班次（`flights.candidates`），已预订的高亮；不再只给单一航班。
- **酒店按"片区 + 价位"推荐**：综合景点位置推荐 1–2 个住宿片区，每片区给经济/中档/高端选项（`hotelAreas`），不直接指定单一酒店。
- **行前须知区块**（`preTrip`）：按出发季节定制的天气/穿搭/台风提醒、支付方式、必备 App、购票时机。
- **卡片增强**：景点 slot 增加营业时间、休息日、门票参考价、点到点交通（方式/票价/耗时）、时令限定活动等字段。
- **餐饮升级**：原 `meals.food`（一句话）改为 `dining` 数组，每餐含餐厅、营业时间、必点菜及参考价。
- **实用贴士与天气驱动排程**：全程 `tips` + 当日 `tips`；排程把户外项目放在凉爽时段、午后/极端时段安排室内。
- **单日二选一方案**（`days[].alternatives`）、每日 `weekday`/`theme`。
- **全覆盖免责声明**：免责范围从"机票/酒店"扩展到全部联网信息（天气、餐厅、评分、营业时间、价格、活动等），强调 AI 整理、可能过时、需自行核实。
- **图片稳健性**：图片 URL 统一用 `https://commons.wikimedia.org/wiki/Special:FilePath/<文件名>?width=N` 形式（不手拼哈希直链），且**每个 URL 必须校验返回 200 才使用**；图片用 `object-fit: cover` 防变形。
- **设计步骤解耦（无硬依赖）**：原设计硬调 `frontend-design`，现改为可插拔——优先用 `frontend-design` 或 `huashu-design`（任一已安装），都没有则用 `references/design-guidelines.md` 内置准则。使 skill 可独立分享，不强制先装其他 skill。
- **跨 Agent 支持**：工具名去 Claude 专属化（联网搜索改用通用说法），可安装到 Claude Code（`~/.claude/skills`）与 OpenAI Codex（`~/.codex/skills`）；其他 Agent 的适配方法与通用提示词见 `references/porting-to-other-agents.md`。

> 这些能力均已写入 `page-contract.md` / `research-guide.md` / `SKILL.md`，并在 `samples/` 的香港、深圳样例中体现。
